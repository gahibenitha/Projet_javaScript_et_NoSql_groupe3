[À REMPLIR : Écrire un résumé concis couvrant :]

FitTracker est une application web complète de suivi de fitness permettant aux utilisateurs 
de gérer leurs séances d'entraînement, leurs exercices, et leurs objectifs personnels. 
L'application propose un système d'authentification sécurisé, un catalogue d'exercices 
avec filtrage avancé, la création de séances personnalisées, l'enregistrement des 
entraînements réalisés avec statistiques automatiques, et un suivi des objectifs de fitness.

Technologies utilisées :
  • Frontend : HTML5, CSS3, JavaScript ES6+ (Vanilla)
  • Backend : Node.js, Express.js, MongoDB (Mongoose)
  • Sécurité : JWT (JSON Web Tokens), bcryptjs pour le hachage des mots de passe
  • Outils : Chart.js pour les graphiques, Postman pour les tests API

L'application comprend 6 collections MongoDB interconnectées, un système de rôles 
(user/admin), et des agrégations complexes pour les statistiques. Le projet est 
entièrement fonctionnel et répond à toutes les exigences académiques.

3.1 ARCHITECTURE LOGICIELLE

[À REMPLIR : Expliquer l'architecture 3-tiers]

Notre application suit le modèle 3-tiers :

┌──────────────────────────────────────────────────────────────┐
│                   COUCHE PRÉSENTATION (Frontend)             │
│  HTML/CSS/JavaScript - Navigateur web                        │
│  Pages : Accueil, Login, Dashboard, Exercices, etc.         │
└──────────────────────┬───────────────────────────────────────┘
                       │ HTTP/REST
┌──────────────────────▼───────────────────────────────────────┐
│              COUCHE MÉTIER (Backend - Node.js/Express)       │
│  Contrôleurs : authentification, séances, exercices          │
│  Middleware : authentification JWT, validation               │
│  Routes API : /api/auth, /api/exercices, /api/seances, etc. │
└──────────────────────┬───────────────────────────────────────┘
                       │ Mongoose/MongoDB
┌──────────────────────▼───────────────────────────────────────┐
│           COUCHE DONNÉES (Base de données - MongoDB)         │
│  Collections : User, Seance, Exercice, DetailSeance, etc.   │
└──────────────────────────────────────────────────────────────┘

Avantages de cette architecture :
  • Séparation des responsabilités
  • Scalabilité : backend peut être déployé indépendamment
  • Maintenabilité : chaque couche peut être modifiée indépendamment
  • Testabilité : tests possibles à chaque niveau

JUSTIFICATION DES RELATIONS

Embarquement vs Référence :
  • Utilisé Référence (ref) pour : Utilisateur → Seance, Exercice, Historique, Objectif
    (Raison : chaque collection a son cycle de vie indépendant)
  
  • Utilisé Référence pour : DetailSeance → Seance, Exercice
    (Raison : permet réutilisation et modification indépendante)

Avantages NoSQL :
  • Flexibilité : ajout de nouveaux champs sans migration de schéma
  • Scalabilité : sharding possible sur une clé pertinente
  • Performance : récupération rapide avec lookups (agrégations)


3.3 AUTHENTIFICATION ET SÉCURITÉ

[À REMPLIR : Expliquer le flux d'authentification]

Flux d'inscription :
1. Utilisateur soumet formulaire : nom, email, mot de passe
2. Backend valide les données (email format, mot de passe min 6 caractères)
3. Vérification qu'email n'existe pas déjà
4. Mot de passe hashé avec bcryptjs (salt rounds = 10)
5. Utilisateur créé en base de données
6. JWT généré avec payload : { id, role }
7. Token retourné au frontend

Flux de connexion :
1. Utilisateur soumet : email, mot de passe
2. Backend cherche l'utilisateur par email
3. Mot de passe saisi comparé avec hash stocké (bcryptjs.compare)
4. Si valide : JWT généré et retourné
5. Si invalide : erreur 400

Utilisation du JWT :
1. Frontend stocke token en localStorage
2. À chaque requête API, token inclus en header : "Authorization: Bearer <token>"
3. Middleware backend valide le token
4. Si valide : ajoute req.utilisateur avec l'ID et rôle
5. Si invalide : retourne erreur 401

Routes protégées :
  • GET /api/auth/profil - nécessite authentification
  • PUT /api/auth/profil - nécessite authentification
  • GET /api/seances - nécessite authentification (filtrée par utilisateur)
  • POST /api/seances - nécessite authentification
  • etc.

4.2 IMPLÉMENTATION DE FONCTIONNALITÉS CLÉS

[À REMPLIR : Mettre les vrais extraits de code]

Exemple 1 : Authentification et Middleware JWT

// Backend - middleware/auth.js
const jwt = require('jsonwebtoken');

const proteger = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ succes: false, message: 'Token manquant' });
  
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.utilisateur = decoded;
    next();
  } catch {
    res.status(401).json({ succes: false, message: 'Token invalide' });
  }
};

module.exports = { proteger };


Exemple 2 : Agrégation MongoDB - Statistiques hebdomadaires

// Backend - historiqueController.js
exports.statistiques = async (req, res) => {
  try {
    const stats = await Historique.aggregate([
      { $match: { utilisateur: new mongoose.Types.ObjectId(req.utilisateur.id) } },
      {
        $group: {
          _id: { semaine: { $week: '$dateEntrainement' }, annee: { $year: '$dateEntrainement' } },
          totalSeances: { $sum: 1 },
          totalCalories: { $sum: '$calories' },
          totalDuree: { $sum: '$dureeReelle' }
        }
      },
      { $sort: { '_id.annee': -1, '_id.semaine': -1 } },
      { $limit: 12 }
    ]);
    res.json({ succes: true, data: stats });
  } catch (err) {
    res.status(500).json({ succes: false, message: err.message });
  }
};

Cette agrégation regroupe les entraînements par semaine pour afficher les 
statistiques du dashboard.


Exemple 3 : Frontend - Appel API et affichage

// Frontend - js/dashboard.js
async function chargerStatistiques() {
  const token = localStorage.getItem('token');
  const response = await fetch('http://localhost:5000/api/historique/statistiques', {
    method: 'GET',
    headers: { 'Authorization': 'Bearer ' + token }
  });
  
  const data = await response.json();
  if (data.succes) {
    afficherGraphique(data.data);
  } else {
    alert('Erreur : ' + data.message);
  }
}

function afficherGraphique(stats) {
  const ctx = document.getElementById('graphique').getContext('2d');
  new Chart(ctx, {
    type: 'bar',
    data: {
      labels: stats.map(s => 'Semaine ' + s._id.semaine),
      datasets: [{
        label: 'Séances',
        data: stats.map(s => s.totalSeances)
      }, {
        label: 'Calories',
        data: stats.map(s => s.totalCalories)
      }]
    }
  });
}

6. DIFFICULTÉS RENCONTRÉES ET SOLUTIONS
───────────────────────────────────────────────────────────────────────────────

[À REMPLIR : Décrire 3+ difficultés avec solutions]

DIFFICULTÉ 1 : Port 5000 déjà utilisé
───────────────────────────────────────
Problème :
  Au lancement du serveur : "Error: listen EADDRINUSE: address already in use :::5000"
  Le port 5000 était utilisé par un ancien processus Node.js qui ne s'était pas 
  arrêté correctement.

Cause :
  Un serveur précédent n'avait pas été tué proprement.

Solution :
  1. Identifier le PID : netstat -ano | findstr :5000
  2. Terminer le processus : taskkill /PID 9904 /F
  3. Relancer le serveur

Impact :
  ✓ Résolu - serveur fonctionne normalement


DIFFICULTÉ 2 : Navigation entre pages HTML
─────────────────────────────────────────
Problème :
  Les clics sur les menus affichaient l'erreur "Cannot GET /dashboard.html"
  Les liens pointaient vers les fichiers HTML au lieu des routes backend.

Cause :
  Les éléments HTML avaient href="dashboard.html" au lieu de href="/dashboard"
  Le backend n'était pas configuré pour servir les fichiers HTML par les routes.

Solution :
  1. Ajouter les routes dans server.js :
     app.get('/dashboard', (req, res) => { 
       res.sendFile(path.join(__dirname, '../frontend/pages/dashboard.html')); 
     });
  
  2. Corriger les liens HTML :
     <a href="/dashboard">Dashboard</a>
  
  3. Corriger les redirections JavaScript :
     window.location.href = '/dashboard' (au lieu de 'dashboard.html')

Impact :
  ✓ Résolu - navigation complète entre les pages fonctionne


DIFFICULTÉ 3 : Ordre des routes Express avec paramètres dynamiques
──────────────────────────────────────────────────────────────
Problème :
  La route GET /api/historique/statistiques retournait erreur 404
  car Express interprétait "statistiques" comme un paramètre :id

Cause :
  L'ordre des routes : d'abord /:id puis /statistiques
  Express compare les routes dans l'ordre et :id match avant /statistiques

Solution :
  Placer les routes spécifiques AVANT les routes paramétrées :
  
  // Avant : ❌ ne fonctionne pas
  router.get('/:id', ...)
  router.get('/statistiques', ...)
  
  // Après : ✓ fonctionne
  router.get('/statistiques', ...)
  router.get('/exercices-populaires', ...)
  router.get('/:id', ...)

Impact :
  ✓ Résolu - routes spécifiques maintenant accessibles

ANNEXE B : GUIDE D'INSTALLATION ET D'EXÉCUTION

Prérequis :
  • Node.js v22+ (ou compatible)
  • MongoDB v5+ (local ou cloud)
  • npm (fourni avec Node.js)

Étapes d'installation :

1. Cloner le projet
   $ git clone <URL_REPO>
   $ cd fitness-app

2. Installer les dépendances
   $ npm install

3. Configurer les variables d'environnement
   Créer fichier .env à la racine :
   PORT=5000
   MONGO_URI=mongodb://localhost:27017/fitness-app
   JWT_SECRET=votre_secret_tres_securise

4. Démarrer MongoDB (si local)
   $ mongod

5. Lancer le serveur
   $ npm run dev
   
   Le serveur démarre sur http://localhost:5000

6. Ouvrir le navigateur
   $ http://localhost:5000
   
   Accès aux pages : /, /login, /register, /dashboard, /exercices, etc.

Tester les APIs avec Postman :

1. Inscription
   POST http://localhost:5000/api/auth/inscription
   Body (JSON) :
   {
     "nom": "Jean Dupont",
     "email": "jean@example.com",
     "motDePasse": "Password123",
     "age": 30,
     "poids": 75,
     "taille": 180
   }

2. Connexion
   POST http://localhost:5000/api/auth/connexion
   Body (JSON) :
   {
     "email": "jean@example.com",
     "motDePasse": "Password123"
   }
   (Réponse contient un token à utiliser pour les requêtes suivantes)

3. Créer une séance (requiert token)
   POST http://localhost:5000/api/seances
   Headers :
     Authorization: Bearer <TOKEN>
   Body (JSON) :
   {
     "nom": "Full Body Monday",
     "description": "Séance complète",
     "dureeEstimee": 60,
     "estPublique": false
   }
